# Vokda Hub Architecture Spec

**Status:** Planning
**Scope:** Extending Vokda from a voice catalog into the full TTS hub — adding news, model tracking, learning content, and the structural changes to support it.

---

## 1. The Two-Track Model

Vokda runs two parallel content tracks that are architecturally distinct but visually unified:

| Track | Content | Tooling | Audience |
|-------|---------|---------|----------|
| **Discovery** | Voices, collections, audition | SvelteKit + Amplify | Find & use voices |
| **Hub** | News, model profiles, guides, reviews | Jekyll + JSON feeds | Stay current, learn, decide |

The discovery track already exists. This spec plans the hub track and how the two connect.

---

## 2. Jekyll: Separate Repo (Recommendation)

### Decision: Yes, separate repo.

**Repo:** `iksnae/vokda-hub`

**Why separate:**

1. **Different toolchain.** Jekyll is Ruby. Vokda is Node/TypeScript. Mixing them in one monorepo creates dependency friction, longer build times, and contributor confusion.

2. **Different publishing cadence.** News and guides ship continuously — an article can go from draft to live without touching the app. Feature work has its own cycle. Separate repos = separate deploys with no coupling.

3. **Different contributor profile.** Editorial and curatorial work (writing guides, tracking model releases) doesn't require access to the full app codebase. A content repo is lower-stakes and more approachable.

4. **Different CI needs.** Jekyll builds are deterministic static-site pipelines. App builds need TypeScript checks, Amplify integration, preview envs, etc. Keeping CI clean on both sides is easier separate.

5. **Flexibility.** If Jekyll turns out to be wrong for a particular content type, swapping it out doesn't touch the discovery app.

**What stays in the main `vokda` repo:**
- The SvelteKit app consumes a JSON feed generated by Jekyll
- The app renders a lightweight "Latest from the Hub" panel on the home page
- Deep links between the two (voice detail → provider profile, news item → voice card)

## 4. Domain & Routing Strategy

### Phase 1 (Now): Subdomain

```
vokda.iknsae.com          → SvelteKit app (Amplify)
hub.vokda.iknsae.com      → Jekyll site (GitHub Pages or Amplify static)
```

Keep it simple. Ship content fast. The subdomain lets Jekyll go live without any CloudFront reconfiguration.

### Phase 2 (Later): Unified domain under /hub

Once Vokda has a proper `.com` domain, consolidate:

```
vokda.com                 → SvelteKit app (all /voices, /collections, /hub routes)
vokda.com/hub/*           → Jekyll output (served via CloudFront behavior routing)
vokda.com/hub/news        → Jekyll news archive
vokda.com/hub/guides      → Jekyll guides index
```

CloudFront behaviors route by path prefix — `/hub/*` hits the Jekyll S3 origin, everything else hits Amplify. No iframe, no redirect. One domain, two origins, seamless nav.

### Jekyll hosting options (Phase 1)

| Option | Effort | Cost | Custom domain |
|--------|--------|------|---------------|
| GitHub Pages | Minimal | Free | Yes (CNAME) |
| Amplify static | Low | ~Free | Yes |
| Netlify | Low | Free | Yes |

**Recommendation:** GitHub Pages for Phase 1. It's zero-config for a Jekyll repo, and `hub.vokda.iknsae.com` via CNAME takes 5 minutes to set up.

## 5. Integration: How SvelteKit Consumes Jekyll

### JSON Feed Pattern

Jekyll generates `/api/news.json` at build time:

```json
{
  "generated": "2026-03-05T22:00:00Z",
  "items": [
    {
      "title": "Cartesia releases Sonic 2",
      "date": "2026-03-05",
      "slug": "cartesia-sonic-2-release",
      "url": "https://hub.vokda.iknsae.com/news/cartesia-sonic-2-release",
      "summary": "Cartesia's second-generation Sonic model...",
      "tags": ["release", "cartesia"],
      "providerId": "cartesia"
    }
  ]
}
```

SvelteKit `/hub/+page.ts` fetches this at request time (or build time for static pre-render):

```ts
export const load = async ({ fetch }) => {
  const res = await fetch('https://hub.vokda.iknsae.com/api/news.json');
  const feed = await res.json();
  return { news: feed.items };
};
```

### Deep Links

**Vokda → Hub:** Voice detail pages link to provider/model profiles.
```
"About ElevenLabs →" links to hub.vokda.iknsae.com/providers/elevenlabs
```

**Hub → Vokda:** News articles and model profiles deep-link into voice cards.
```
"Hear Cartesia Sonic 2 →" links to vokda.iknsae.com/voices/{voiceId}
```

Provider `id` in voices.json matches the Jekyll `_providers/` slug — this is the cross-reference key.

## 6. Navigation Changes

Current nav: `Explore | Collections | [Sign in]`

Proposed nav: `Voices | Hub | Collections | [Sign in]`

- **Voices** → `/` (the catalog — rename from "Explore" for clarity)
- **Hub** → `/hub` (in-app hub home, pulls from Jekyll feed; links out to full hub site)
- **Collections** → `/collections` (unchanged)

The in-app `/hub` page is a lightweight aggregator — latest news, featured guide, top model releases. It's the teaser. The full depth lives on `hub.vokda.iknsae.com`.

## 7. Amplify Data Schema Extensions

Jekyll handles editorial content (no auth, no persistence needed). Amplify handles user-generated and personalized data.

### New models needed in `amplify/data/resource.ts`:

```ts
// User review of a voice
VoiceReview: a.model({
  voiceId: a.string().required(),
  provider: a.string().required(),
  naturalness: a.integer(),      // 1-5
  expressiveness: a.integer(),   // 1-5
  consistency: a.integer(),      // 1-5
  useCaseFit: a.string(),        // e.g. "narration", "assistant"
  body: a.string(),
  createdAtIso: a.string().required(),
  updatedAtIso: a.string()
}).authorization((allow) => [
  allow.owner(),
  allow.groups(['curator', 'admin']),
  allow.publicApiKey().to(['read'])
]),

// Bookmarked hub articles (so users can save news items)
SavedArticle: a.model({
  slug: a.string().required(),
  url: a.string().required(),
  title: a.string().required(),
  savedAtIso: a.string().required()
}).authorization((allow) => [allow.owner()]),
```

### Existing models to extend:

```ts
// Collection: add visibility: 'public' to allow sharing
visibility: a.enum(['private', 'team', 'public']),
```

## 9. Rollout Sequence

### Step 1: Create `iksnae/vokda-hub` repo
- Bootstrap Jekyll with basic config
- Set up `_posts`, `_providers`, `_data` directories
- Configure `api/news.json` Liquid template
- Deploy to GitHub Pages at `hub.vokda.iknsae.com`

### Step 2: Seed content
- Write 3-5 initial news posts (recent model releases, TTS industry updates)
- Create provider profiles for: ElevenLabs, OpenAI, AWS Polly, Azure Speech, Google Cloud TTS, Kokoro, Qwen3
- Write 1-2 guides (e.g., "Choosing a voice for your AI agent", "Cloud vs open-source TTS")

### Step 3: Wire SvelteKit
- Add `/hub` route that fetches Jekyll JSON feed
- Rename "Explore" → "Voices" in nav, add "Hub" link
- Add "Provider profile →" links on voice detail pages

### Step 4: Schema & reviews
- Add `VoiceReview` and `SavedArticle` to Amplify data schema
- Build basic review UI on voice detail page (curator+ to post)

### Step 5: CloudFront unification (later)
- When real domain is acquired
- Route `/hub/*` to Jekyll S3, all else to Amplify
- Update all internal links
